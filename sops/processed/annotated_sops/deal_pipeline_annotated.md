## Step 1: Understanding Deal Types

### Original SOP Instruction

> - Deals are split into Main Deals and Sub Deals.
>   - Main Deals represent the overall client engagement.
>   - Sub Deals represent individual courses, services, or projects.

---

### Step 1.1: Understand Main/Sub Deals

**Status:** 🔴 MISSING

**CRM Reality:**
- No "Main Deal" or "Sub Deal" field exists in CRM
- No hierarchy or parent-child relationship between Deals
- Each Deal is independent

**⚠️ Gap Identified:**
- SOP assumes Deal hierarchy that doesn't exist
- Users may search for non-existent fields

**Recommendation:**
- Either add Parent_Deal lookup field to CRM
- Or update SOP to reflect flat Deal structure

### Step 3.3: Move to Quoted

**Status:** ❌ CONTRADICTS

**CRM Reality:**

## Corrected SOP Summary

### Actual CRM Pipeline Stages

```
Qualification → Negotiation/Review → Ready to Quote → [Awaiting ZipPay] →
Awaiting PO → Awaiting Invoice Payment → Closed Won/Closed Lost
```

### Recommended Updated Procedure

**Step 1: Understanding Deals**
- Each Deal is independent (no Main/Sub hierarchy)
- One Deal per student per course

**Step 2: Creating Deals**
1. Create Deal from Course record
2. Set Courseaa, Contact_Name, Account_Name
3. Deal Name auto-generates - do not enter manually
4. Amount auto-calculates from Course pricing

**Step 3: Stage Progression**
1. **Qualification** (initial) → Verify details, set Course_Start_Date
2. **Negotiation/Review** → Chase requirements
3. **Ready to Quote** → Quote auto-creates
4. Back to **Negotiation/Review** → Chase acceptance
5. **Closed Won** → Via Quote acceptance or PO entry (not manual)
6. **Closed Lost** → Enter Loss_Reason

**Step 4: Pipeline Hygiene**
1. Review Deals weekly using saved filter views
2. Update stages to reflect actual status
3. PO reminders fire automatically at 28/14/5/1 days before course
4. Enter Loss_Reason immediately when deals are lost
